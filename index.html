Connect Tiara to your GitHub

Your board lives safely in your own GitHub account. Connect it once and you never download anything again — just visit this page.

  1. Open github.com/settings/tokens → Generate new token (classic).
  2. Name it "Tiara", then tick just two boxes: repo and project.
    repo = read/write your issues + board data · project = drive the GitHub boards
  3. Click Generate token and copy it (starts with ghp_…).
  4. Paste it below and press Connect.

Change process · the full journey of one change

1 The one rule

Nothing is finished until two things are true:

  • the gate comes back green (make verify)
  • the change works in the app on a phone (watched on a real device)

2 The gate

Every change goes through one command, make verify. It runs four quick checks in order, cheapest first, so a problem surfaces in two seconds instead of after a long build:

  • Tidy (swiftformat): the code is re-formatted to the house style, so spacing is never an argument.
  • Rules (swiftlint --strict): the things formatting can't check: no banned patterns, no stray colours or magic numbers, no forbidden shortcuts.
  • Build (xcodegen + xcodebuild): the app compiles.
  • Test (Swift Testing + XCUITest): the tests run.

All four passing is the moment a change is allowed to exist. If any one fails, the change is simply not done yet.

3 The safety nets underneath

Three guards keep the gate honest:

  • The push leash (guard-bash.sh + the .verify-pass fingerprint): the automated helper cannot send code to GitHub unless the gate passed on exactly that code. The leash applies only to code edited and pushed by AI, never to a person at the keyboard.
  • The canary (scripts/gate-canary.sh): every so often a test is deliberately broken to confirm the gate still catches it. A gate that cannot fail is worthless, so it is proven, on purpose.
  • No surprises from time or chance (an injected Now clock + SeededRandomNumberGenerator): tests run on a fixed clock and fixed dice, so a test never passes one day and fails the next for no reason.

4 The deeper checks, run on demand

Beyond the quick gate sit heavier checks, each on its own, run when the relevant area changes:

  • Concurrency (make tsan, Thread Sanitizer): hunts for the bug where two things happen at once and collide.
  • Accessibility (make a11y: a WCAG contrast test + performAccessibilityAudit): confirms text has enough contrast and the screen works with assistive tools. Text must also grow when the system font size is raised (Dynamic Type); that one is a firm rule for every app. These findings are a to-do list, not a blocked build (advisory vs. blocker).
  • Design snapshots (make snapshot, ImageRenderer pixel checks): confirm the colours and components still match how they were defined, so nothing quietly drifts to grey.
  • Deep correctness (property tests + API contract tests): extra tests that hammer the important logic across many combinations, plus a check that any API still answers in the shape the app expects.
  • Speed (make perf, XCTest measure): measures how fast the heavy paths run, so a slowdown shows as a trend.

5 Why almost nothing runs in the cloud automatically

Quick tests can run locally and frequently. The cloud (Xcode Cloud) has one job: a clean-room second opinion, building the app from scratch the way a brand-new computer would, to catch the "worked on one machine" surprises. It runs only when triggered by an increase in build number in the config file (CURRENT_PROJECT_VERSION in Config/Version.xcconfig).

6 Shipping to testers

Changing one number, the build number (CURRENT_PROJECT_VERSION in Config/Version.xcconfig), and pushing is the only thing that triggers a release. That, and nothing else, tells the cloud to build the app, sign it, and hand it to TestFlight (Xcode Cloud archive plus TestFlight deploy). Ordinary commits never ship. Shipping is a deliberate, one-number act, and not every app needs it: it is opt-in (marked by release.yml).

7 How every app is born with all of this

None of this is set up by hand per app. One command (newapp, from the factory template) stamps out a new app that already carries the gate, the safety nets, the deeper checks, the dev-and-real split (separate DEV and PROD schemes, bundle ids, and CloudKit containers), and the written-down rules (the ADRs). Every app inherits the whole machine on day one, so quality is never bolted on later during a scary audit. The Compliance tab shows, at a glance, how far along each app is across all of it.
